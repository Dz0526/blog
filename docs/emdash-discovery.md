## 2. `astro.config.mjs` shape

```js
import cloudflare from "@astrojs/cloudflare";
import react from "@astrojs/react";
import { d1, r2, sandbox } from "@emdash-cms/cloudflare";
import { formsPlugin } from "@emdash-cms/plugin-forms";
import webhookNotifier from "@emdash-cms/plugin-webhook-notifier";
import { defineConfig, fontProviders } from "astro/config";
import emdash from "emdash/astro";

export default defineConfig({
	output: "server",
	adapter: cloudflare(),
	image: {
		layout: "constrained",
		responsiveStyles: true,
	},
	integrations: [
		react(),
		emdash({
			database: d1({ binding: "DB", session: "auto" }),
			storage: r2({ binding: "MEDIA" }),
			plugins: [formsPlugin()],
			sandboxed: [webhookNotifier],
			sandboxRunner: sandbox(),
			marketplace: "https://marketplace.emdashcms.com",
		}),
	],
	fonts: [
		{
			provider: fontProviders.google(),
			name: "Inter",
			cssVariable: "--font-sans",
			weights: [400, 500, 600, 700],
			fallbacks: ["sans-serif"],
		},
		{
			provider: fontProviders.google(),
			name: "JetBrains Mono",
			cssVariable: "--font-mono",
			weights: [400, 500],
			fallbacks: ["monospace"],
		},
	],
	devToolbar: { enabled: false },
});
```

Key points:
- `output: "server"` — required; no static generation for CMS content.
- D1 binding name: `DB` (matched in `wrangler.jsonc`).
- R2 binding name: `MEDIA`.
- Plugin sandboxing via `sandboxRunner: sandbox()` and `sandboxed: [webhookNotifier]`.
- Marketplace URL registered for the admin plugin browser.

## 6. `src/content/` inventory

**No `src/content/` directory exists.** EmDash uses a "live content collection" pattern:

- `src/live.config.ts` registers the `_emdash` collection using `emdashLoader()` from `emdash/runtime`.
- Content is fetched from the D1 database at request time (runtime-generated), not from the filesystem.
- No markdown files are read by the Astro content layer.

```ts
// src/live.config.ts
import { defineLiveCollection } from "astro:content";
import { emdashLoader } from "emdash/runtime";

export const collections = {
  _emdash: defineLiveCollection({ loader: emdashLoader() }),
};
```

The `_posts/` and `_nippo/` directories at the repo root are legacy Next.js markdown content and will be migrated to the D1 database during Phase 6.

       search    Full-text search across content
     taxonomy    Manage taxonomies and terms
         menu    Manage menus
       plugin    Manage plugins

Key CLI notes:
- **Local commands** (work on SQLite, no server needed): `init`, `dev`, `seed`, `export-seed`, `secrets`.
- **Remote commands** (talk to a running server): `types`, `login`, `content`, `schema`, `media`, `search`, `taxonomy`, `menu`, `plugin`.
- Dev server auto-runs migrations and seeds an empty DB on first request — no explicit `emdash init` required for local dev.
- `emdash dev` is an alternative to `astro dev` that also manages the local database.
- Type generation: `pnpm exec emdash types` hits the running server and updates `emdash-env.d.ts`.

## 8. MCP endpoint

Found in `node_modules/emdash/dist/astro/routes/api/mcp.mjs` and `middleware/auth.mjs`.

**URL path:** `/_emdash/api/mcp`

**HTTP methods:**
- `POST /_emdash/api/mcp` — JSON-RPC tool calls (main entry point)
- `GET  /_emdash/api/mcp` — SSE stream (not used in stateless mode)
- `DELETE /_emdash/api/mcp` — Session close (not used in stateless mode)

**Auth mechanism:** Bearer token via `Authorization: Bearer <token>` header. Unauthorized requests receive HTTP 401 with `WWW-Authenticate: Bearer resource_metadata="<origin>/.well-known/oauth-protected-resource"`. The OAuth protected resource metadata endpoint is at `/.well-known/oauth-protected-resource`.

The route is injected automatically by the `emdash()` Astro integration (controlled by `resolvedConfig.mcp` — defaults to `true`). No manual route registration needed.

The `.mcp.json` at the project root points to the EmDash documentation MCP server (`https://docs.emdashcms.com/mcp`), which is a separate cloud-hosted docs server — not the local site's content MCP endpoint.

## 10. Content type creation flow

### Admin UI flow

1. Navigate to `/_emdash/admin` and log in with passkey.
2. Go to **Schema** in the left sidebar.
3. Click **New Collection** to create a new content type.
4. Fill in: slug (e.g. `posts`), label (e.g. `Posts`), singular label, and select supports (drafts, revisions, search, SEO).
5. Add fields: click **Add Field**, choose type (string, portableText, image, etc.), set slug, label, required flag.
6. Save the collection — the migration runs automatically in the background.

### Programmatic / CLI options

**Via seed file** (recommended for project setup): Define collections in `seed/seed.json` under the `"collections"` key. The seed is applied automatically on the first request to an empty database. Running `pnpm exec emdash seed --file seed/seed.json` also applies it to a running local instance.

**Via CLI against a running instance:**
```bash
# List collections
pnpm exec emdash schema list

# Show a collection
pnpm exec emdash schema show posts

# Create content programmatically
pnpm exec emdash content create posts --data '{"title": "Hello", "status": "published"}'
```

**Via export/import:** Run `pnpm exec emdash export-seed` against a configured instance to export the current schema as a seed file — useful for snapshotting admin-created schemas into version control.
